# Docker

É um **gerenciador de containers** que encapsula nossa aplicação dentro dele.

Não importa o ambiente em que você esteja rodando, o **Docker garante que a aplicação rodará sempre da mesma maneira**, **pois o container** **além de conter sua aplicação, também contém tudo que é necessário para que o sistema funcione**.

Em virtualizações convencionais temos um software instalado na máquina host que irá gerenciar as máquinas virtuais (ex.: VirtualBox, VMWare).

Para cada máquina virtual temos uma instalação completa do S.O que queremos virtualizar, além de ter o próprio hardware virtualizado.

Se por exemplo eu precisar de uma biblioteca comum para todas as máquinas virtuais, preciso instalar em cada uma delas.

**O Docker utiliza uma abordagem diferente, o conceito de Containers**, logo o software empacotado sempre será executado da mesma forma em qualquer plataforma em que o Docker seja executado.

## O que é Docker

Além de ser um **gerenciador de containers**, é uma plataforma que **facilita a criação e administração de ambientes isolados** possibilitando o empacotamento de uma aplicação ou ambiente dentro de um container, se tornando portátil para qualquer outro host que contenha o Docker instalado.

A ideia do Docker é subir apenas uma máquina ao invés de várias e nessa única máquina, você pode rodar várias aplicações sem que haja conflito entre elas.

## Principais diferenças entre Docker e Virtual machines

### Docker

* Utiliza containers que isolam processos e recursos;
* Compartilha o mesmo kernel do SO hospedeiro;
* Menor consumo de recursos por conta do item acima;
* Inicialização rápida;
* Contém todos os componentes, bibliotecas e dependências necessárias para a aplicação funcionar;
* Fácil implementação em diferentes ambientes;
* Isolamento de processos;
* Utiliza arquivos Dockerfile para definir a configuração da imagem;
* Utiliza Docker Engine para criar, gerenciar e executar containers.

### Virtual Machine

* Emula todo o hardware, incluindo SO para cada virtual machine;
* Requer mais recursos devido à emulação de hardware separado para cada virtual machine;
* Inicialização mais lenta;
* Podem ser mais difíceis de migrar entre diferentes ambientes devido às diferenças de SO;
* Não compartilha recursos de maneira direta para outras virtual machines;
* Utiliza hipervisores para criar e gerenciar virtual machines.

## Containers

**Um container é um ambiente isolado que contém um conjunto de processos que são executados a partir de uma imagem e uma imagem fornece todos os arquivos necessários.** Eles compartilham o mesmo kernel e isolam os processos da aplicação do restante do sistema.

**Um contêiner executa uma imagem. O contêiner é uma instância em execução da imagem.**

**Um contêiner é uma unidade de software que inclui tudo o que é necessário para executar um aplicativo, incluindo o código, bibliotecas, dependências e configurações.**

Pode ser iniciado, pausado, reiniciado e interrompido.

## Imagens

**Uma imagem é um pacote de sistema de arquivos que inclui todas as dependências necessárias para executar um processo/aplicação** e isso abrange arquivos de biblioteca, itens no sistema de arquivos, pacotes instalados, recursos disponíveis, processos em execução e módulos do kernel.

O Docker opera com o conceito de imagens; em outras palavras, para lançar um container, o Docker deve possuir a imagem no hospedeiro, já que essa **imagem** **serve como um modelo para o container.**

**Uma imagem Docker é um pacote leve e autossuficiente que abriga todas as informações requeridas para criar um contêiner.**

As imagens de contêiner precisam estar disponíveis localmente ou armazenadas e mantidas em um repositório de imagens. Um repositório de imagens é simplesmente um serviço público ou privado onde as imagens podem ser armazenadas, buscadas e recuperadas.

Uma imagem pode ser compartilhada, versionada e utilizada para gerar diversos contêineres.

## Volumes

**Os volumes permitem que os dados e alterações sejam compartilhados e persistidos entre containers e o host.**

São uma maneira flexível de lidar com a persistência de dados, permitindo que você armazene informações que não são parte do sistema de arquivos do container, mas que podem ser compartilhadas e mantidas de maneira independente, facilitando a administração de dados em containers.

Devemos pensar em volumes como **pastas que podem ser usadas para guardar informações dos containers, como arquivos, registros de banco de dados e elas mantêm os dados mesmo após reiniciar os containers, ou seja, quando derrubarmos um container, os dados que que armazenamos nestes volumes não irão desaparecer, irão estar guardados.**

**Se você não utilizarmos volumes, toda vez que subirmos um novo contêiner com a mesma imagem, ele começará do zero, como se fosse a primeira vez**.

## Dockerfile

**É um arquivo de texto utilizado para definir e criar uma imagem Docker personalizada com base nas nossas necessidades pessoais.**

Nele podemos inserir e encontrar uma série de instruções que especificam como construir o ambiente de execução para um container.

O Dockerfile **descreve quais arquivos, bibliotecas, configurações e comandos são necessários para criar uma imagem que possa ser utilizada para iniciar containers com uma configuração específica.**

Dockerfile é uma escolha pessoal do que uma imagem Docker irá ter.

## docker-compose

O arquivo docker-compose.yaml **é um arquivo de configuração utilizado para definir a estrutura, configuração e a execução de vários containers do Docker** como uma aplicação única.

Ele simplifica a criação, gestão e manutenção de ambientes complexos, **permitindo que você defina várias partes da sua aplicação como containers, redes, volumes, variáveis de ambiente, etc em um único arquivo.**

Também automatiza a configuração e orquestração de containers interconectados (com dependências de outros containers) e assim simplifica o processo de desenvolvimento, teste e implantação de aplicações compostas por vários serviços.

## Alguns comandos úteis

* **Expand**

  ## Docker ps (process status)

  É utilizado para listar os containers em execução no Docker e suas informações básicas

  ```bash
  docker ps
  
  ```

  ## Docker ps -a (all)

  É utilizado para mostrar todos os containers, inclusive aqueles que não estão em execução no momento, ou seja, podem estar também parados, pausados ou com execução finalizada.

  Útil para rastrear todos os containers e ajudar na depuração, limpeza e entendimento do histórico de atividade dos containers

  ```bash
  docker ps -a
  
  ```

  ## Docker run

  Utilizado para criar e iniciar um novo container a partir de uma imagem, ou seja, executa aplicativos ou serviços em containers isolados

  ```bash
  docker run <options> image
  
  ```

  ## Docker run -d (detached)

  Utilizado para executar um container em segundo plano, ou seja, sua saída não é exibida no terminal

  ```bash
  docker run -d image
  
  ```

  ## Docker run -i (interactive)

  Executa um container no modo interativo, ou seja, podemos interagir com o container normalmente através de seu terminal.

  Normalmente utilizamos este comando junto com a flag -t (terminal) e isso permite com que possamos executar comandos dentro do container em execução

  ```bash
  docker run -it image
  
  ```

  ## Docker run -p (map system ports)

  É utilizado para mapear uma porta do host para uma porta do container ao executar um container no Docker.

  Permite que você especifique como as portas do container se comunicarão com as portas do host.

  **Exemplo:** `docker run -p 80:8000`

  Neste caso estaríamos dizendo que 80 seria a porta no sistema do host e 8000 a porta dentro do container que gostaríamos de expor.

  Com isso significa que ao acessar a porta 80 no host, estaríamos redirecionando o tráfego para a porta 8000 dentro do container.

  **Exemplo**: Se executarmos uma aplicação Django dentro de um container rodando o comando `python3 manage.py runserver 127.0.0.1:8080` e executássemos o container utilizando o comando `docker run -p 8000:8080 django-app` poderíamos acessar esta aplicação através do nosso host através de uma url `http://127.0.0.1:8000`

  Essa funcionalidade é especialmente **útil para expor serviços da aplicação em contêineres para o mundo exterior, permitindo que você acesse a aplicação por meio do navegador ou de outras ferramentas, como se ela estivesse sendo executada diretamente no host.**

  ```bash
  docker run -p <host_port>:<container_port> image
  
  ```

  ## Docker exec

  Utilizado para executar um comando dentro de um container que já está em execução, permitindo que você interaja com um container em funcionamento, como se estivesse abrindo um terminal dentro dele

  ```bash
  docker exec -it container /bin/bash
  
  ```

  ## Docker -v (volume)

  Utilizado para criar e associar um volume entre o host e o container na sua criação.

  Permite o compartilhamento para a não perda de dados entre o SO do host e o sistemas de arquivos do container.

  ```bash
  docker run -v <caminho_no_host>:<caminho_no_contêiner imagem
  docker run -v /caminho/local:/var/lib/postgresql/data postgres
  ```

  ## Docker build

  Utilizado para criar uma imagem Docker a partir de um conjunto de instruções contidas em um arquivo Dockerfile que define as as etapas e configurações necessárias para construir uma imagem Docker.

  O processo de construção (build) executa essas etapas em ordem, gerando uma imagem que pode ser usada para criar e executar containers.

  **Exemplo:** `docker build -t <nome_da_imagem>:<tag> <caminho_do_contexto>`

  ```bash
  docker build -t meu_app:1.0 ./meu_app
  
  ```
  * `t meu-app:1.0` define um nome (meu_app) e uma tag (1.0) para a imagem que será criada.

  `./meu_app` especifica o caminho para o diretório onde se encontra o Dockerfile;
