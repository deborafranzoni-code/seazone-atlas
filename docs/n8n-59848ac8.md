# n8n

> **Esta migração é executada ao vivo com Claude Code. Todos os manifests, ExternalSecrets e configurações são gerados e aplicados durante a janela de manutenção. O engenheiro valida e aprova cada passo; Claude Code executa.**

**App:** N8N Production **Namespaces GKE:** `n8n` (postgres + redis + editor) e `prd-n8n` (stateless) **Namespace EKS destino:** `prd-n8n` **Complexidade:** Alta **Janela estimada:** 30–45 minutos **Domínios:** `automation.seazone.com.br`, `webhook-automation.seazone.com.br`

## 1. Arquitetura Atual (GKE)

### Cluster GKE: `cluster-tools-prod-gke` (us-central1-a, `https://34.60.64.40`)

```
Namespace: n8n  (infra stateful)
├── postgres-n8n            StatefulSet    PVC 5Gi (dados do banco N8N)
├── redis-n8n               Deployment     PVC 1Gi (filas Bull + cache)
└── n8n-editor              Deployment     PVC 1Gi (community packages, .n8n/)

Namespace: prd-n8n  (aplicação, 100% stateless)
├── n8n-editor (1 pod)      conecta no postgres e redis do namespace n8n
├── n8n-mcp    (1 pod)      conecta no mesmo postgres/redis
├── n8n-webhooks (6 pods)   HPA 2→6
├── n8n-workers  (6 pods)   HPA 2→6
└── n8n-backup (CronJob)    daily 03:00 → GCS bucket szn-n8n-prd-backup
```

**Dependências externas GKE:**

* Redis em `10.23.1.3:6379` (DB 1)
* PostgreSQL em host configurado via Secret `n8n-secrets` (chaves `DB_POSTGRESDB_HOST`, `DB_POSTGRESDB_DATABASE`, etc.)
* Workload Identity GKE: `n8n-prd-sa@tools-440117.iam.gserviceaccount.com` (usado pelo initContainer `npm-wrapper` para acessar Google Artifact Registry `us-central1-npm.pkg.dev/tools-440117/n8n-npm`)
* GCS bucket: `szn-n8n-prd-backup` (backup de workflows/credenciais)

**Application ArgoCD atual:**

* `destination.server: https://34.60.64.40`
* `syncPolicy.automated.prune: false` / `selfHeal: false` (sync manual)
* ExternalSecret: lê de `/sre/n8n/prd/` no AWS SSM via `ClusterSecretStore: aws-parameter-store-global`

## 2. Arquitetura Destino (EKS)

### Cluster EKS: `general-cluster` (sa-east-1, `https://kubernetes.default.svc`)

```
Namespace: prd-n8n  (aplicação stateless — igual ao GKE)
├── n8n-editor (1 pod)
├── n8n-mcp    (1 pod)
├── n8n-webhooks (HPA 2→6)
├── n8n-workers  (HPA 2→6)
└── n8n-backup (CronJob) → S3 ou mantém GCS (ver seção 5)

Banco de Dados: RDS tools-postgres (db.t4g.small, sa-east-1)
└── database: n8n_prd  (schema/database dedicado, novo)

Redis: redis-n8n-prd (já existe no EKS)
└── argocd/applications/databases/redis-n8n-prd/
    IP interno EKS (service ClusterIP redis-n8n-prd)
    Port 6379, DB 0 (rever se conflito com outros apps)
```

**Mudanças em relação ao GKE:**

| Item | GKE | EKS |
|----|----|----|
| Banco de dados | postgres StatefulSet in-cluster (5Gi PVC) | RDS `tools-postgres` (managed) |
| Redis | Deployment in-cluster (1Gi PVC) | `redis-n8n-prd` in-cluster (EKS) |
| Ingress | Traefik GKE | Traefik EKS (NLB sa-east-1) |
| Workload Identity | GKE SA + GCP IAM | IRSA (AWS IAM Role for Service Account) |
| npm packages | Google Artifact Registry (GCP) | Remover dependência ou migrar para ECR/CodeArtifact |
| Backup destino | GCS `szn-n8n-prd-backup` | Manter GCS (bucket público) ou migrar para S3 |
| ServiceAccount annotation | `iam.gke.io/gcp-service-account` | `eks.amazonaws.com/role-arn` |

## 4. Backup Antes da Migração

### 4.1 Disparar CronJob de Backup Manualmente

O CronJob `n8n-backup` no namespace `prd-n8n` (GKE) exporta workflows e credenciais para GCS. Claude Code executa no início da janela:

```bash
# Criar Job único a partir do CronJob

kubectl create job --from=cronjob/n8n-backup n8n-backup-pre-migration \
  -n prd-n8n --context=<gke-context>

# Acompanhar execução

kubectl logs -f job/n8n-backup-pre-migration -n prd-n8n --context=<gke-context>

# Confirmar arquivos no GCS

gcloud storage ls gs://szn-n8n-prd-backup/workflows/ --project=tools-440117

gcloud storage ls gs://szn-n8n-prd-backup/credentials/ --project=tools-440117
```

### 4.2 Dump do Banco PostgreSQL (GKE)

O banco está no StatefulSet `postgres-n8n` no namespace `n8n` do GKE.

```bash
# Variáveis

GKE_CONTEXT="<seu-gke-context>"
N8N_PG_POD=$(kubectl get pod -n n8n -l app=postgres-n8n \
  -o jsonpath='{.items[0].metadata.name}' --context=$GKE_CONTEXT)

echo "Pod postgres GKE: $N8N_PG_POD"

# Dump completo (substitua DB_NAME e DB_USER pelo valor real dos secrets GKE)
kubectl exec -n n8n $N8N_PG_POD --context=$GKE_CONTEXT -- \
  pg_dump -U <DB_POSTGRESDB_USER> -d <DB_POSTGRESDB_DATABASE> \
  --format=custom --compress=9 --verbose \
  -f /tmp/n8n_prd_backup_$(date +%Y%m%d_%H%M).dump

# Copiar dump para local

kubectl cp n8n/$N8N_PG_POD:/tmp/n8n_prd_backup_*.dump \
  ./n8n_prd_backup_$(date +%Y%m%d_%H%M).dump \
  --context=$GKE_CONTEXT

# Verificar tamanho e integridade

ls -lh n8n_prd_backup_*.dump

pg_restore --list n8n_prd_backup_*.dump | head -20
```

### 4.3 Export Redis (opcional)

O Redis do N8N armazena filas temporárias (Bull). As execuções em andamento serão perdidas na migração — isso é esperado. Para referência:

```bash
# Listar chaves Bull no Redis GKE (apenas auditoria, não é necessário migrar)
REDIS_POD=$(kubectl get pod -n n8n -l app=redis-n8n \
  -o jsonpath='{.items[0].metadata.name}' --context=$GKE_CONTEXT)
kubectl exec -n n8n $REDIS_POD --context=$GKE_CONTEXT -- \
  redis-cli -n 1 KEYS "bull:*" | wc -l
```

## 5. Passo a Passo da Migração

### Etapa 1 — Preparar RDS no EKS

```sql
-- Conectar no RDS tools-postgres (via kubectl exec no pod postgres-tools no EKS, ou psql direto)
-- Criar database e usuário para N8N PRD

CREATE DATABASE n8n_prd;
CREATE USER n8n_prd WITH PASSWORD '<senha-forte>';
GRANT ALL PRIVILEGES ON DATABASE n8n_prd TO n8n_prd;
\c n8n_prd

GRANT ALL ON SCHEMA public TO n8n_prd;
```

### Etapa 2 — Restore do Dump no RDS

```bash

EKS_CONTEXT="<seu-eks-context>"
RDS_HOST="<endpoint-rds-tools-postgres>.sa-east-1.rds.amazonaws.com"

# Opção A: via pod temporário no EKS com acesso ao RDS

kubectl run pg-restore-tmp --image=postgres:17 --restart=Never \
  -n tools --context=$EKS_CONTEXT \
  -- sleep 3600

kubectl cp ./n8n_prd_backup_*.dump \
  tools/pg-restore-tmp:/tmp/n8n.dump \
  --context=$EKS_CONTEXT

kubectl exec -n tools pg-restore-tmp --context=$EKS_CONTEXT -- \
  pg_restore \
    -h $RDS_HOST \
    -U n8n_prd \
    -d n8n_prd \
    --no-owner --no-privileges \
    --verbose \
    /tmp/n8n.dump

# Limpar pod temporário

kubectl delete pod pg-restore-tmp -n tools --context=$EKS_CONTEXT
```

### Etapa 3 — Validar Restore

```bash
# Conectar no RDS e verificar tabelas e contagens

kubectl run pg-check-tmp --image=postgres:17 --restart=Never \
  -n tools --context=$EKS_CONTEXT \
  -- sleep 600

kubectl exec -it -n tools pg-check-tmp --context=$EKS_CONTEXT -- \
  psql -h $RDS_HOST -U n8n_prd -d n8n_prd -c "
    SELECT tablename, pg_size_pretty(pg_total_relation_size(tablename::text))
    FROM pg_tables
    WHERE schemaname = 'public'
    ORDER BY pg_total_relation_size(tablename::text) DESC;
  "

# Verificar tabelas principais do N8N

kubectl exec -it -n tools pg-check-tmp --context=$EKS_CONTEXT -- \
  psql -h $RDS_HOST -U n8n_prd -d n8n_prd -c "
    SELECT 'execution_entity' as tabela, count(*) FROM execution_entity
    UNION ALL
    SELECT 'workflow_entity', count(*) FROM workflow_entity
    UNION ALL
    SELECT 'credentials_entity', count(*) FROM credentials_entity;
  "

kubectl delete pod pg-check-tmp -n tools --context=$EKS_CONTEXT
```

### Etapa 4 — Atualizar Secrets no AWS SSM

```bash
# Atualizar host do banco para o RDS

aws ssm put-parameter \
  --name "/sre/n8n/prd/DB_POSTGRESDB_HOST" \
  --value "$RDS_HOST" \
  --type SecureString --overwrite \
  --region sa-east-1

aws ssm put-parameter \
  --name "/sre/n8n/prd/DB_POSTGRESDB_DATABASE" \
  --value "n8n_prd" \
  --type SecureString --overwrite \
  --region sa-east-1

aws ssm put-parameter \
  --name "/sre/n8n/prd/DB_POSTGRESDB_USER" \
  --value "n8n_prd" \
  --type SecureString --overwrite \
  --region sa-east-1

aws ssm put-parameter \
  --name "/sre/n8n/prd/DB_POSTGRESDB_PASSWORD" \
  --value "<senha>" \
  --type SecureString --overwrite \
  --region sa-east-1

# Confirmar que N8N_ENCRYPTION_KEY está idêntica ao GKE (NÃO alterar)
aws ssm get-parameter \
  --name "/sre/n8n/prd/N8N_ENCRYPTION_KEY" \
  --with-decryption --region sa-east-1 \
  --query 'Parameter.Value' --output text
```

### Etapa 6 — Atualizar values-n8n.yaml para EKS

Claude Code aplica as seguintes alterações em `argocd/applications/n8n/prd/values-n8n.yaml`:

```yaml
# Remover ServiceAccount GKE Workload Identity:
serviceAccount:
  create: true
  name: "n8n-sa"
  annotations:
    # REMOVER: iam.gke.io/gcp-service-account: n8n-prd-sa@tools-440117.iam.gserviceaccount.com
    # ADICIONAR (se necessário para S3/AWS):
    eks.amazonaws.com/role-arn: "arn:aws:iam::711387131913:role/n8n-prd-role"

# Atualizar Redis host para o redis-n8n-prd no EKS:
config:
  - name: QUEUE_BULL_REDIS_HOST
    value: "redis-n8n-prd.tools.svc.cluster.local"  # service name EKS
  - name: QUEUE_BULL_REDIS_PORT
    value: "6379"
  - name: QUEUE_BULL_REDIS_DB
    value: "0"

# Remover initContainer npm-wrapper (dependência GCP):
editor:
  initContainers: []   # remover bloco inteiro
  extraEnvs: []        # remover PATH customizado
  extraVolumeMounts: []
  extraVolumes: []

# Adicionar nodeSelector/tolerations para Karpenter EKS:

  nodeSelector:
    node-pool: general-karpenter-prod
  tolerations: []
```

> **Atenção npm-wrapper:** O initContainer que busca token OAuth2 do metadata server GCP (`metadata.google.internal`) irá falhar no EKS. Se a funcionalidade de community packages precisar de registry privado, reconfigurá-la usando CodeArtifact AWS ou remover o wrapper (pacotes serão buscados do [npmjs.com](http://npmjs.com) público).

### Etapa 7 — Atualizar IngressRoute para EKS

O arquivo `argocd/applications/n8n/prd/ingressroute.yaml` não precisa de alterações de conteúdo. O Traefik no EKS irá processar as mesmas regras. Garantir que os domínios DNS apontem para o NLB do EKS após a validação.

### Etapa 8 — Atualizar CronJob de Backup para EKS

O CronJob atual usa `gcloud storage` com Workload Identity GCP. Para o EKS, duas opções:

**Opção A (recomendada — manter GCS):** Usar chave de serviço GCP via Secret em vez de Workload Identity.

```yaml
# No CronJob, substituir serviceAccountName por um que use IRSA para S3
# OU usar uma GCP Service Account Key como Secret convencional

containers:
- name: gcloud-upload
  env:
  - name: GOOGLE_APPLICATION_CREDENTIALS
    value: /gcp-key/key.json
  volumeMounts:
  - name: gcp-key
    mountPath: /gcp-key
    readOnly: true

volumes:
- name: gcp-key
  secret:
    secretName: n8n-gcp-sa-key   # Secret com a chave JSON da SA
```

**Opção B — Migrar backup para S3:**

```yaml

containers:
- name: aws-s3-upload
  image: amazon/aws-cli:latest
  command:
  - /bin/sh
  - -c
  - |
    set -e
    DATE="$(date +%Y-%m-%d)"
    aws s3 cp /backup/credentials.json s3://szn-n8n-prd-backup/credentials/credentials-backup-${DATE}.json
    aws s3 cp /backup/workflows.json s3://szn-n8n-prd-backup/workflows/workflows-backup-${DATE}.json
    echo "Backup S3 concluído"
```

## 6. Janela de Manutenção

| Item | Detalhe |
|----|----|
| **Dia/horário** | Sábado, 02h00–03h00 (horário de Brasília) |
| **Duração estimada** | 30–45 minutos |
| **Downtime esperado** | 15–25 min (do scale-down no GKE até DNS propagar no EKS) |
| **Comunicação** | 48h antes no canal #tech-infra e notificação para usuários de automation |
| **Equipe mínima** | 1 SRE + 1 usuário de negócio para validar workflows |

### Sequência na Janela de Manutenção

```
T+00:00  Claude Code dispara backup manual (CronJob job) e aguarda conclusão

T+05:00  Claude Code escala workers/webhooks para 0 no GKE (drenar execuções)
T+07:00  Claude Code executa pg_dump do postgres-n8n (GKE) — ~5min

T+12:00  Claude Code cria database n8n_prd no RDS + pg_restore — ~5-10min

T+20:00  Validar contagens e integridade do banco restaurado

T+22:00  Claude Code atualiza secrets SSM (DB_POSTGRESDB_HOST, etc.)
T+23:00  Claude Code edita application.yaml (destination → EKS) + values-n8n.yaml e commita

T+24:00  ArgoCD sync prd-n8n — ~3min

T+27:00  Validar pods no EKS (Running) — ~2min

T+29:00  Teste funcional: login, workflow manual, webhook

T+33:00  Claude Code redireciona DNS no Cloudflare

T+38:00  Confirmar DNS propagado (~5min TTL) + teste final pelos usuários

T+43:00  Manter GKE de pé por 48h (rollback disponível)
```

## 7. Rollback Plan

Se algo falhar após o sync no EKS e antes do redirecionamento de DNS:

```bash
# 1. Reverter application.yaml para destination GKE

git revert HEAD  # ou editar manualmente e commitar

# 2. Forçar sync do ArgoCD apontando de volta para GKE

argocd app sync prd-n8n --server argocd.seazone.com.br

# 3. Escalar workers/webhooks de volta no GKE

kubectl scale deployment n8n-workers --replicas=5 -n prd-n8n --context=$GKE_CONTEXT

kubectl scale deployment n8n-webhooks --replicas=6 -n prd-n8n --context=$GKE_CONTEXT
```

Se o DNS já foi redirecionado e a aplicação no EKS está com problemas:

```bash
# 1. Reverter DNS imediatamente no Cloudflare (CNAME de volta para NLB GKE)
# TTL curto (60s) durante a janela facilita rollback rápido

# 2. Reverter application.yaml e forçar sync ArgoCD

# 3. Investigar logs no EKS:
kubectl logs -n prd-n8n -l app.kubernetes.io/name=n8n --context=$EKS_CONTEXT --tail=100
```

**Janela de rollback:** O cluster GKE deve permanecer ativo e com os pods rodando por **48h** após a migração. Só desligar o GKE após confirmação dos usuários de negócio.

  name: prd-n8n

  finalizers:
    - resources-finalizer.argocd.argoproj.io
